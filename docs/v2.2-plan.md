# ovnflow v2.2.0 plan

v2.2.0 starts the native flow-programming and SD-WAN foundation layer.

## Native OpenFlow

- Provide a pure Go OpenFlow 1.5/1.3 codec and thin client.
- Support hello/version negotiation, features request/reply, flow-mod,
  multipart flow dump envelopes, protocol errors, OXM basic matches, output
  actions, and set-field actions.
- Expose `client.OpenFlow()` with bridge-scoped fluent flow builders and
  cookie ownership boundaries.
- Support both caller-provided OpenFlow endpoints and SDK bridge-controller
  auto-configuration through OVSDB.
- Do not shell out to `ovs-ofctl`.

## SD-WAN Foundation

- Model open Site, Link, Policy, L2/L3 overlay, explicit Partial Mesh,
  Hub-Spoke, Full Mesh, WireGuard, Geneve, and VXLAN primitives.
- Keep business concepts such as organization, department, user, scheduler,
  approval, billing, and HA election outside the SDK.
- Provide `Plan`, `DryRun`, `Apply`, `Get`, and `Delete` over a pluggable
  backend.
- Use an in-memory backend for tests and controller planning; production
  callers can provide a backend that applies Linux routes, WireGuard, OVS
  tunnels, and OpenFlow rules.

## Acceptance

- Ordinary `go test ./...` must not require OVS, root, WireGuard, or a live
  OpenFlow endpoint.
- Unit tests must cover OpenFlow codec bytes, match round trips, handshake over
  `net.Pipe`, cookie ownership, SD-WAN Partial Mesh, L2 OpenFlow planning, and
  API reuse through `Client.SDWAN()`.
- `tools/sdkcheck` must compile and exercise the public OpenFlow/SD-WAN API
  surface without requiring a live OpenFlow endpoint.

## Boundaries

- Partial Mesh uses caller-provided `SDWANLink` objects. Full Mesh is the
  topology that auto-generates all site pairs; Hub-Spoke auto-generates links
  from sites marked `Role: "hub"`, `Transit`, or `Relay` to ordinary spokes.
- The default `Client.SDWAN()` backend is in-memory so controllers can plan,
  dry-run, and test without requiring Linux routes, WireGuard, OVS tunnels, or
  an OpenFlow endpoint. Production callers can pass `Config.SDWAN`, call
  `UseSDWANBackend`, or use `NewSDWANClient(customBackend)`.
- Live OpenFlow bridge-controller integration and production SD-WAN backend
  implementations are future hardening work; v2.2.0 provides the protocol,
  fluent API, planning model, and backend contract.
