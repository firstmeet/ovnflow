# ovnflow v2.3.0 plan

v2.3.0 turns the v2.2 OpenFlow and SD-WAN foundation into a production-facing
runtime layer while keeping business scheduling, HA election, tenant models,
and private controller loops outside the SDK.

## Delivered Scope

- Live OpenFlow integration against a real OVS bridge controller endpoint:
  configure a bridge, connect with the native OpenFlow client, add a flow, dump
  flow stats, delete the named flow, and verify the cookie disappears.
- Docker/CI now starts `ovs-vswitchd` and exposes `6653` in addition to OVSDB
  `6640` and OVN DB `6641/6642`.
- SD-WAN sites and links now carry optional attributes and disabled-link
  semantics while preserving the existing `SDWANBackend` interface.
- Linux-only `sdwanlinux` backend implements real backend hooks for:
  - WireGuard interface and peer commands through `ip` and `wg`.
  - Linux route and policy-rule installation through `ip route` and `ip rule`.
  - OVS Geneve/VXLAN tunnel rows through `client.LocalOVS()` wrappers.
  - Optional L2 OpenFlow rule installation through `client.OpenFlow()`.
- Open SD-WAN agent/control-plane primitives:
  agent identity, capabilities, heartbeat, assignment, assignment ACK/result,
  and an in-memory control plane for tests and simple controllers.

## API Boundaries

- Existing v2.2 APIs remain source-compatible.
- `SDWANBackend` is not expanded with required methods; production backends are
  injected through `Config.SDWAN`, `Client.UseSDWANBackend`, or
  `NewSDWANClient(customBackend)`.
- The default `Client.SDWAN()` backend remains in-memory.
- Linux execution lives in the `sdwanlinux` subpackage so Windows builds and
  normal tests do not require root, WireGuard, OVS, or Linux tools.
- WireGuard private keys and preshared keys are passed by file path attributes;
  key material is not copied into status, plan output, or logs.
- Destructive OVS tunnel cleanup verifies ovnflow SD-WAN ownership markers
  before deleting tunnel ports.

## Test Gates

- Normal local validation:
  - `go test -count=1 ./...`
  - `go vet ./...`
  - `go -C tools/sdkcheck test -count=1 ./...`
- Linux compile validation for the production backend:
  - `GOOS=linux GOARCH=amd64 go test -c ./sdwanlinux`
- OVSDB/OVN integration:
  - `OVNFLOW_REQUIRE_INTEGRATION=1 go test -tags=integration -count=1 -v ./...`
- Live OpenFlow integration:
  - `OVNFLOW_OPENFLOW_ADDR=tcp:127.0.0.1:6653`
  - `OVNFLOW_OPENFLOW_CHECKS=1`
  - `go test -tags=integration -count=1 -v -run 'TestIntegrationOpenFlow(Endpoint|FluentEndpoint)Lifecycle' ./...`
- SD-WAN OVS tunnel integration:
  - `OVNFLOW_SDWAN_BACKEND_CHECKS=1`
  - `OVNFLOW_OVS_TUNNEL_CHECKS=1`
  - `OVNFLOW_OPENFLOW_CHECKS=1` when also validating the OpenFlow hook
  - `go test -tags=integration -count=1 -v -run 'TestIntegrationSDWAN(OVSTunnel|OpenFlowHook)Lifecycle' ./sdwanlinux`
- Privileged SD-WAN WireGuard/Linux route integration:
  - `OVNFLOW_SDWAN_BACKEND_CHECKS=1`
  - `OVNFLOW_SDWAN_PRIVILEGED_CHECKS=1`
  - `OVNFLOW_WIREGUARD_CHECKS=1`
  - `OVNFLOW_LINUX_ROUTE_CHECKS=1`
  - `sudo -E go test -tags=integration -count=1 -v -run TestIntegrationSDWANWireGuardLinuxRouteLifecycle ./sdwanlinux`

## Still Out Of Scope

- Tenant, organization, department, user, quota, approval, and private database
  models.
- HA election, leader selection, keepalived-style failover, VIP movement, and
  node scheduling.
- Full SD-WAN controller policy optimization. The SDK exposes neutral runtime
  primitives; the caller decides assignment, placement, and failover policy.
