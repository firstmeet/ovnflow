# ovnflow v2.1 implementation plan

## Goal

`v2.1` extends the open networking foundation without embedding private
platform concepts. Business objects such as tenant, organization, department,
user, quota, approval, billing, placement, and HA election stay in the caller's
control plane. Callers map those concepts to `OwnerRef`, `Labels`, and regular
metadata.

## Scope

### IPAM helpers

- Pure Go IPv4 CIDR helpers for planning virtual-network addresses.
- Gateway selection, reserved/excluded addresses, explicit allocation,
  automatic allocation, release, containment, availability, and overlap checks.
- No database, daemon, lease service, global lock, scheduler, or controller
  loop.
- IPv6 returns a typed unsupported error until the API is broadened.

### Service and load-balancer intent

- Open service abstraction over OVN `Load_Balancer`.
- VIPs, backend sets, protocol, optional router attachment, owner, and labels.
- Reuse the existing handwritten `LoadBalancer` fluent API instead of adding
  another OVSDB transaction path.
- Do not model Kubernetes Service, service registry, tenant service catalog, or
  authorization.

### QoS policy intent

- Open OVN QoS abstraction over direction, priority, match, bandwidth, DSCP,
  mark, optional logical-switch attachment, owner, and labels.
- Reuse the existing `QoS` fluent API.
- OVS QoS/Queue intent remains future work.

### Status aggregation

- Read-only typed summaries for network, workload path, and provider-network
  status.
- Aggregate existing `Get`, `Doctor`, and `AuditOwnership` data.
- Missing backends degrade into typed findings instead of panics.

### Safe cleanup and adopt/import planning

- Read-only cleanup/adopt plans built from ownership audit data.
- Report candidates, ownership state, risks, and suggested actions.
- No default deletion of unmanaged resources. Any future execution path must be
  explicit, owned-only, and marker-checked.

## Agent Split

- Lead/integrator: owns public naming, common metadata semantics, docs, final
  merge, tests, PR, and release.
- IPAM worker: owns `ipam.go` and `ipam_test.go`.
- Service/LB worker: owns service intent files and tests.
- QoS worker: owns QoS intent files and tests.
- Status/Cleanup worker: owns status and cleanup plan files and tests.
- Plan reviewer: read-only scope and API review.
- Code reviewer: read-only final diff review.
- Test reviewer: read-only test and CI review.

Workers must prefer new files, avoid duplicated function/type names, and avoid
editing large existing files unless a thin public entry point is required.

## Acceptance Criteria

- `go test -count=1 ./...` passes without OVN, OVS, WSL, Docker, or root.
- `go vet ./...` passes.
- New APIs return ovnflow typed errors for validation, unsupported families,
  unavailable backends, conflicts, and not-found cases.
- Owner and label metadata remain neutral and do not introduce private business
  entities.
- Cleanup/adopt planning is read-only by default and never suggests deleting
  unmanaged resources without explicit risk.
- README, roadmap, and release notes describe the new open primitives and the
  out-of-scope boundaries.
- A PR is opened from `codex/v2-next-network-foundation`.
- Release is performed only after the branch has passing local checks and CI.

## Deferred

- Persistent IPAM service and distributed lease coordination.
- OVS QoS/Queue high-level intent.
- Packet-level LinuxRouter NAT/firewall integration tests across live
  namespaces.
- Multi-WAN policy-routing helpers.
- HA, keepalived, leader election, failover, VIP movement, and node scheduling.
