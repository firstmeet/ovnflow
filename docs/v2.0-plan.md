# v2.0.0 multi-agent pre-tag acceptance checklist

This document records the `ovnflow` v2.0.0 implementation scope, agent model,
release gates, and evidence that must be collected before and after the
`v2.0.0` tag is published. The goal is to move beyond table-level fluent APIs
and add open, composable network intent primitives without embedding private
business models.

## Summary

v2.0.0 uses multiple agents for planning, review, and test design, but only one
coding path may modify files. This avoids merge conflicts while still keeping
independent review pressure on scope, safety, and CI evidence.

v2.0.0 keeps the v1 OVSDB API stable and adds higher-level primitives:

- ownership metadata and labels,
- virtual networks and workload attachment,
- logical-switch DNS with multiple IPs per domain,
- platform-neutral security policy,
- diff, dry-run, reconcile, status, inspect, and safe cleanup,
- Linux namespace router with DNSMasq, NAT, firewall, object-style Get/Apply/Patch,
  and Linux-only privileged tests.

The foundation library must not include organization, department, user, billing,
quota, HA election, node scheduling, keepalived, approval workflow, or private
database concepts.

## Agent model

| Agent | Write access | Responsibility |
| --- | --- | --- |
| Lead/Integrator | No production-code writes | Owns branch, CI, release gates, conflict resolution, and final acceptance. Repository mutations still flow through the sole coding path unless the user explicitly redirects. |
| Code Worker | Yes, sole code/docs writer | Implements all production code, tests, docs, and workflow changes assigned by Lead. No other agent may edit files. |
| Plan Reviewer | No | Reviews scope for openness, missing primitives, and private-platform leakage. |
| Code Reviewer | No | Reviews architecture, cleanup safety, concurrency, build tags, errors, and API stability. |
| Test Reviewer | No | Reviews unit, integration, privileged Linux, WSL, Docker, and GitHub CI coverage. |
| Release Reviewer | No | Reviews changelog, release notes, tag workflow, module resolution, and final `v2.0.0` publication evidence. |

The Code Worker is the sole code/docs writer for the v2.0.0 implementation.
Reviewer agents must be read-only and should report findings with file paths and
concrete risk levels.

## Delivered implementation scope

v2.0.0 implements these open primitives:

- `OwnerRef`, `Labels`, and reserved `external_ids` helpers for ownership,
  grouping, adoption checks, cleanup, and reconciliation.
- `VirtualNetwork` desired-state APIs over logical switches, subnets, DHCP/DNS,
  default ACLs, metadata, Get/Apply/Patch, Diff, DryRun, Status, and Delete.
- `LogicalSwitchDNS` APIs that attach OVN DNS records to logical switches and
  support one domain mapped to multiple IP addresses.
- `WorkloadAttachment` APIs that keep OVN logical switch ports and local OVS
  Port/Interface metadata aligned, including `iface-id`, MAC/IP, owner labels,
  status, detach, and safe shared-interface handling.
- `ProviderNetwork` APIs that keep an OVN logical switch localnet port and
  Open_vSwitch `ovn-bridge-mappings` aligned for physical network access,
  without owning or deleting the physical bridge by default.
- `SecurityPolicy` APIs over ACL, Port Group, and Address Set, using neutral
  allow/drop/CIDR/port/established semantics.
- `LinuxRouter` as a Linux-only module for managed network namespaces, LAN/WAN
  interfaces, static or DHCP WAN addressing, routes, DNSMasq, NAT, firewall,
  detailed Get status, Apply, Patch, module-specific Ensure/Delete, and safe
  cleanup.

LinuxRouter NAT must include:

- fixed-source SNAT,
- interface MASQUERADE,
- inbound DNAT port forwarding,
- destination-address mapping,
- one-rule update and delete by stable rule name,
- single-WAN/single-LAN inference only when unambiguous,
- typed ambiguous-interface errors for multiple candidates.

LinuxRouter DNSMasq must include:

- DHCP ranges,
- DNS forwarding,
- static leases,
- host records,
- multiple A records for the same domain,
- pidfile, leasefile, config file, and process status handling.

## Architecture decisions

- Keep root package OVSDB APIs stable. Add platform-neutral public types in the
  root package only when they are useful outside LinuxRouter.
- Put host networking in a Linux-only package or module boundary. Real host
  networking implementations must use `//go:build linux`; non-Linux stubs must
  compile and return typed unsupported errors.
- Do not reuse the OVSDB executor for host commands. Add a separate host
  command/filesystem/process abstraction for `ip`, `dnsmasq`, `nft`, `iptables`,
  `conntrack`, and namespace operations.
- Hide NAT backend details behind a backend interface. Prefer nftables when
  available, support iptables, and never let callers depend on backend command
  syntax.
- Use SDK-owned markers everywhere: `external_ids`, namespace names/labels,
  OVS port metadata, DNSMasq config headers, pidfile metadata, nftables comments
  or chains, and iptables comments.
- Default cleanup must refuse unmanaged resources. Any force mode still must be
  constrained to resources with explicit `ovnflow` ownership markers.
- `Apply` means full desired-state reconciliation for the addressed object.
  `Patch` and module-specific builders mean incremental change and must not
  delete unrelated modules.
- Object updates must guard against concurrent stale writes with an observed
  hash, generation, or resource version.

## Test and CI gates

Ordinary tests must never require OVN, OVS, Docker, WSL, root, dnsmasq,
nftables, iptables, or CAP_NET_ADMIN.

Release-blocking test layers:

- Windows unit: compile and test all platform-neutral APIs and non-Linux stubs.
- Linux unit and race: normal tests, Linux-only command rendering, fake
  host-executor tests, staticcheck, and govulncheck.
- OVSDB integration: Docker OVN/OVS endpoints for `VirtualNetwork`,
  `LogicalSwitchDNS`, `WorkloadAttachment`, `ProviderNetwork`,
  `SecurityPolicy`, and Diagnostics.
- v2 mutation gate: real create, read, ensure, update, watch, delete, safe
  cleanup, and orphan/reference assertions for v2 OVSDB-backed APIs.
- LinuxRouter privileged gate: root/CAP_NET_ADMIN tests for namespace
  lifecycle, runtime observation, dnsmasq process/readback, multi-address DNS
  records, SNAT, MASQUERADE, DNAT, port forwarding, destination mapping,
  firewall rule installation, owned-rule readback, and cleanup boundaries across
  `auto`, `nftables`, and `iptables` backends.
- WSL manual/extended gate: same endpoint-driven integration style as v1, plus
  optional LinuxRouter checks when privileged WSL is available.

Suggested environment gates:

- `OVNFLOW_V2_SCHEMA_CHECKS=1`
- `OVNFLOW_V2_MUTATION_CHECKS=1`
- `OVNFLOW_LINUX_ROUTER_CHECKS=1`
- `OVNFLOW_NAT_BACKEND=auto|nftables|iptables`

GitHub CI must run all release-blocking tests before `v2.0.0` is tagged. The
release workflow includes unit, vet, staticcheck, govulncheck, race, examples,
bench, OVSDB integration, v2 mutation, and LinuxRouter privileged matrix
checks.

## Pre-tag execution record

1. The roadmap and this plan were committed as the v2 planning baseline.
2. The `codex/v2.0.0` implementation branch was used for v2 work.
3. A single coding path implemented production code, tests, docs, and workflow
   changes. Reviewer agents remained read-only and reported findings back to
   the lead.
4. The interface-freeze scope was documented in `docs/interface-freeze.md`
   before host-networking side effects were expanded.
5. The OVSDB-backed v2 intent APIs were implemented with ownership guards,
   diff/dry-run/reconcile surfaces, typed get/inspect paths, and integration
   tests.
6. The LinuxRouter object model, host executor boundary, dnsmasq support, NAT
   backends, firewall rules, Get/Apply/Patch, and privileged tests were
   implemented.
7. Local checks, Docker integration, mutation gates, GitHub push CI, and manual
   privileged LinuxRouter matrix runs must be green on the final pre-tag SHA.
8. The changelog and release notes are finalized before the annotated
   `v2.0.0` tag is pushed.
9. The release workflow must complete successfully after the tag push.
10. Post-release verification must prove the GitHub Release, Go module
    resolution, and pkg.go.dev page for `v2.0.0`.

## Acceptance criteria

v2.0.0 is complete only when current evidence proves:

- v1 public APIs remain source compatible.
- All v2 public APIs have unit tests and examples.
- OVSDB-backed v2 APIs pass Docker integration and mutation checks.
- LinuxRouter passes privileged real networking tests for both supported NAT
  backends or has documented backend-specific availability checks.
- Safe cleanup proves it does not delete unmanaged resources.
- `Get` returns detailed spec/status for v2 objects, including LinuxRouter NAT,
  DNSMasq, routes, interfaces, firewall, process state, and installed-rule
  observations.
- GitHub CI and release workflow are green.
- `v2.0.0` GitHub Release, `go list -m github.com/firstmeet/ovnflow/v2@v2.0.0`,
  and pkg.go.dev resolve.

## v2.x hardening backlog

These checks are intentionally outside the v2.0.0 release-blocking gate and are
tracked for later hardening:

- Packet-level NAT and firewall translation tests across realistic LAN/WAN
  topologies.
- OVS port movement stress tests for LinuxRouter LAN/WAN attachments.
- DHCP client lease negotiation tests for WAN interfaces in a privileged lab.
- Longer dnsmasq lifecycle and reload tests under repeated Apply/Patch/Delete.
- Large watch lifecycle and multi-version OVN/OVS schema compatibility matrix
  expansion.
