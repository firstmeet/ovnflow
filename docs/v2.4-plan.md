# ovnflow v2.4.0 plan

v2.4.0 adds SD-WAN path-mode compatibility on top of the v2.3 production
backend. The SDK models direct P2P, relay/transit, and auto fallback paths while
leaving health checks, scheduling, HA, and NAT traversal control loops to the
caller.

## Delivered Scope

- `SDWANPathModeDirect`, `SDWANPathModeRelay`, `SDWANPathModeTransit`, and
  `SDWANPathModeAuto`.
- Network-level builder helpers for path mode selection.
- Link-level `PathMode` overrides for explicit direct links in an auto network.
- Hub/relay aware `sdwanlinux` AllowedIPs and Linux route rendering.
- Auto path behavior that does not duplicate a destination CIDR on both direct
  and relay routes while the direct link is enabled.
- Disabled direct links can be cleaned up and their destinations can fall back
  through relay/transit links.

## Boundaries

- Empty path mode remains direct for source compatibility.
- `SDWANBackend` is unchanged.
- `Via` path candidates, health probes, NAT traversal, STUN/TURN/ICE,
  controller scheduling, and HA election remain caller responsibilities.
- Relay eligibility uses existing `Role: "hub"`, `Transit`, and `Relay` site
  fields.

## Test Gates

- `go test -count=1 ./...`
- `go vet ./...`
- `go -C tools/sdkcheck test -count=1 ./...`
- `GOOS=linux GOARCH=amd64 go test -c ./sdwanlinux`
- Linux-only `sdwanlinux` unit tests for relay, auto fallback, and disabled
  direct cleanup.
- Privileged integration verifies auto path route preference: enabled direct
  paths win, disabled direct paths are cleaned up, and relay/transit takes over.
- GitHub PR CI for Windows and Linux unit tests plus Docker integration.
- Release CI includes Windows unit tests, Linux validation, privileged
  LinuxRouter checks, and privileged SD-WAN WireGuard/Linux route checks.
