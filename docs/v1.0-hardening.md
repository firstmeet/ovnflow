# v1.0 hardening and release matrix

This document records the v1.0 release-readiness gates for `ovnflow`.

## Acceptance gates

| Area | v1.0 requirement | Evidence |
| --- | --- | --- |
| Module path | Module path, repository URL, examples, and release tags resolve through `github.com/firstmeet/ovnflow` | `go.mod`, examples, release workflow |
| Public API | v0.1/v0.2 source compatibility is kept and stable entrypoints are documented | `docs/api-stability.md` |
| OVN Northbound | Fluent lifecycle for major NB tables, UUID-safe delete, map/set mutation by default | NB builders and `integration_v1_acceptance_test.go` |
| OVN Southbound | Typed list/get/watch for main runtime tables plus guarded generic writes | `sb.go`, `sb_runtime_test.go`, integration tests |
| Open_vSwitch | Fluent lifecycle for Bridge, Port, Interface, Controller, Manager, Mirror, QoS, Queue, Flow_Table, NetFlow, sFlow, IPFIX, SSL, AutoAttach; Bridge-level helpers attach non-root advanced config rows atomically | `ovs.go`, `ovs_runtime.go`, OVS tests, mutation gate |
| Runtime schema | Required columns fail with `ErrorInvalidSchema`; optional columns degrade without panic; UUID reference columns are classified as scalar, set, or map references before delete cleanup | `schema.go`, `schema_test.go`, WSL/Docker integration |
| Concurrency | Watch fanout/cancel/overflow and initial/live ordering are covered | `fluent_watch_test.go`, Linux race CI |
| Transactions | Production writes use `libovsdb` transactions and preserve external map/set keys | `fluent.go`, NB/OVS builders |
| Deletes | Delete paths select UUIDs first, clean cross-table and same-table UUID set/map references, reject scalar strong-reference deletes with `ErrorConflict`, and avoid deleting shared OVS config, Port, and Interface rows by default | NB/OVS delete tests |
| Tests | Unit, vet, race, examples, Docker integration, and mutation gates pass in CI | `.github/workflows/test.yml` |
| Release | Tag workflow validates unit/vet/race/examples/bench plus Docker integration/mutation before creating a GitHub release | `.github/workflows/release.yml` |

## Remaining non-blocking hardening

- Broaden mutation acceptance to cover every remaining advanced NB table with
  field-level readback. OVS advanced Bridge config now has field-level readback
  and cleanup assertions in the mutation gate.
- Add more public watch lifecycle stress tests around rapid subscribe/cancel
  loops and long-running poller cleanup.
- Fix any future schema-version gaps found by real user OVN/OVS deployments by
  moving fields from required checks to runtime optional handling.

## Release checklist

1. Confirm the working tree is clean on the release branch.
2. Run local validation:
   - `go test -count=1 ./...`
   - `go vet ./...`
   - `go test -run Example ./...`
   - `go test -bench=. -benchmem ./...`
3. Run Linux race validation:
   - `go test -race -count=1 ./...`
4. Run integration validation with reachable OVN/OVS endpoints and require mode:
   - `OVNFLOW_REQUIRE_INTEGRATION=1 go test -tags=integration -count=1 -v ./...`
   - The same command with require mode and missing endpoints must fail, not
     skip; this protects CI and release jobs from false green runs.
5. Run mutation validation before release:
   - `OVNFLOW_REQUIRE_INTEGRATION=1 OVNFLOW_V1_MUTATION_CHECKS=1 go test -tags=integration -count=1 -v -run TestIntegrationV1MutationScenariosAreEnvGated ./...`
6. Push the branch and require GitHub Actions success.
7. Create and push `v1.0.0` only after human release approval.
8. Confirm the GitHub Release, `go list -m github.com/firstmeet/ovnflow@v1.0.0`,
   and the pkg.go.dev module page resolve.
